# Step 4: 개발 로드맵 도출 (roadmap)

## 목적

개발을 어떤 순서로 진행할지를 **도메인 역학(Stage C)에서 도출**한다.
로드맵이 기능 목록의 나열이 되면 실패다 — 각 단계가 "왜 이게 먼저인가"를
도메인 근거로 답할 수 있어야 한다.

## RULES

- 단계(Phase)는 2~4개. 각 단계는 "무엇을 만들/할지 + 왜 이 순서인지 + 이
  단계의 완료 신호"까지만 적는다. **task 분해·기술 설계 금지** — 그건
  design-doc/개발 스킬의 몫이다.
- 개발 항목만 넣지 않는다. 시딩, 첫 사용자 모집, 랜딩 오픈 같은
  **비개발 작업도 같은 로드맵에 넣는다.** 콜드스타트를 통과하는 건 코드가 아니라
  이런 작업들인 경우가 많다.

## 도출 논리 (예시)

Stage C의 답에 따라 순서가 달라진다:

- **콜드스타트 문제가 있다** (볼 콘텐츠/상대방이 필요) → 공급을 채우는 수단이
  Phase 1에 온다. 예: 운영자가 직접 시딩할 수 있는 어드민, 크롤링, 단면 모드
  (혼자 써도 가치 있는 형태)로 시작.
- **획득 경로가 검색/공유다** → 랜딩·공유 가능한 결과물·SEO 구조가 코어 기능과
  같은 Phase에 온다. "일단 기능부터, 마케팅은 나중에"가 통하지 않는 경로다.
- **리텐션 훅이 약하다** → 두 번째 방문 이유(알림, 축적되는 데이터, 정기성)를
  만드는 게 기능 확장보다 앞 Phase에 온다.
- **당장 수익이 필요하다** → 결제 동선이 MVP에 포함된다. 아니면 명시적으로 미룬다.
