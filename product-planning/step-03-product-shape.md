# Step 3: 화면·플로우 확정 (product-shape)

## 목적

`{discovery_log}`를 근거로 **프로덕트의 모양을 제안**한다. 이 step의 산출물이
UI 디자인(피그마, Stitch, 디자인 스킬 등)의 직접 입력이 되므로, 디자이너가
이것만 보고 와이어프레임을 그릴 수 있는 수준이어야 한다. 단, 비주얼 지시는
하지 않는다 — 무엇이 있어야 하는지까지만.

## RULES

- 여기서는 질문이 아니라 **제안**이 기본이다. discovery가 끝났으므로 당신이
  그릴 수 있어야 하고, 사용자는 고치는 역할이다.
- 화면 수는 MVP 기준으로 잡는다. Stage D에서 자른 범위를 벗어나는 화면은
  "이후 단계" 표시를 달아 분리한다.
- Stage C의 획득 경로가 요구하는 화면(랜딩, 공유, 온보딩)을 빠뜨리지 않는다.
  기능 화면만 그리고 유입 화면을 잊는 게 초보 기획의 전형적 실수다.

## INSTRUCTIONS

### 1. 핵심 여정 정의

가장 중요한 사용자 여정 1~2개를 정의한다:
`유입 → ... → 핵심 가치를 경험하는 순간` 의 체인. 핵심 가치 경험까지의
스텝 수를 세어보고, 줄일 수 있으면 줄인 안을 제안한다.

### 2. 화면 목록 작성

여정에 필요한 화면을 나열한다. 화면마다:

- **화면명** / **목적 한 줄**
- **핵심 요소**: 이 화면에 반드시 있어야 하는 것 (목록, 버튼, 입력, 상태 표시 등)
- **주요 상태**: 비어있을 때(콜드스타트!) / 채워졌을 때 / 에러 등, 다르게 보여야 하는 상태
- **이동**: 어느 화면에서 오고 어디로 가는가

### 3. 제시와 확정

여정과 화면 목록을 제시하고 피드백을 받는다. 화면 추가/삭제/병합 의견을 반영해
확정한다.

HALT하고 사용자 확인을 기다린다. 확정된 내용이 `{screens}`가 된다.
