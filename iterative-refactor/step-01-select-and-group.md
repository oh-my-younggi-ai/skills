# Step 1: 리팩토링 파일 선택 및 항목 그룹화

## RULES

- **태스크 파일은 항상 사용자에게 직접 물어서 결정한다. 예외 없음.**
- 이전 대화나 메시지에서 파일이 언급되었더라도 반드시 다시 묻는다.
- 체크박스 `[x]`로 표시된 항목은 이미 완료된 것으로 간주하고 건너뛴다.
- 미완료 항목이 0개면 HALT하고 사용자에게 모두 완료되었음을 알린다.
- **EARLY EXIT** = 이 step을 즉시 중단하고 지정된 파일로 이동한다.


## INSTRUCTIONS

### 1. 태스크 파일 결정

HALT하고 사용자에게 묻는다:

> 리팩토링 체크리스트 파일이 있으신가요?
>
> `[F]` 파일 경로 입력 — 기존 체크리스트 파일을 사용합니다.
> `[D]` 직접 입력 — 리팩토링할 항목을 지금 설명해 주세요.

#### [F] 파일 경로 입력

사용자가 경로를 제공하면 `{task_file}`로 설정하고 단계 2로 진행한다.

#### [D] 직접 입력

사용자가 리팩토링할 내용을 설명하면 아래 순서로 진행한다:

1. 설명을 바탕으로 체크리스트 항목을 추출/정리해 출력한다:

   > 아래 항목으로 체크리스트를 생성하겠습니다:
   > - [ ] #1 {항목 제목}
   >   {설명}
   > - [ ] #2 {항목 제목}
   >   {설명}
   >
   > `[A]` 이대로 진행
   > `[E]` 수정 — 항목을 변경하거나 추가/삭제해 주세요

   HALT하고 사용자 입력을 기다린다.
   - **[E]**: 수정 내용을 반영한 뒤 다시 출력하고 재확인한다.

2. 승인 후 체크리스트를 저장할 경로를 묻는다:

   > 체크리스트 파일을 어디에 저장할까요?
   > (기본값: `{project_root}/refactor-checklist.md`)

   HALT하고 사용자 입력을 기다린다. 사용자가 빈 값을 입력하면 기본값을 사용한다.

3. 확정된 경로에 체크리스트 파일을 Write하고 `{task_file}`로 설정한다.

4. 작성된 파일 내용을 그대로 출력하고 최종 확인을 요청한다:

   > 아래 내용으로 `{task_file}`을 저장했습니다.
   >
   > ```markdown
   > {파일 전체 내용}
   > ```
   >
   > `[A]` 승인 — 이 체크리스트로 리팩토링을 시작합니다.
   > `[E]` 수정 — 파일을 직접 수정한 뒤 알려주세요. 수정 완료 후 다시 읽고 진행합니다.
   > `[X]` 취소

   HALT하고 사용자 입력을 기다린다.
   - **[E]**: 사용자가 완료를 알리면 `{task_file}`을 다시 읽고 단계 4로 돌아가 재확인한다.
   - **[X]**: HALT.

5. 단계 2로 진행한다.


### 2. 기존 progress 파일 확인

`{progress_file}` 경로를 계산한다 (workflow.md의 경로 계산 규칙 참고).

`{progress_file}`이 존재하면:
- frontmatter의 `task_file:` 값이 `{task_file}`과 일치하는지 확인한다.
- 일치하면 frontmatter의 `current_batch_index`와 groups 총 배치 수를 비교한다:

  **진행 중인 세션** (`current_batch_index < groups 총 배치 수`):
  > 이전 진행 상황이 있습니다.
  > 배치 {current_batch_index + 1} / {총 배치 수} 진행 중이었습니다.
  >
  > `[R]` 재개 — 이어서 진행합니다.
  > `[N]` 새로 시작 — 그룹화부터 다시 합니다 (progress 파일 덮어씀).

  **완료된 세션** (`current_batch_index >= groups 총 배치 수`):
  > 이전에 완료된 세션이 있습니다 (`{task_file}`).
  >
  > `[N]` 새로 시작 — 그룹화부터 다시 합니다 (progress 파일 덮어씀).
  > `[X]` 종료

  - **[R]**: frontmatter에서 `{groups}`, `{current_batch_index}` 복원. **EARLY EXIT** → `./step-02-implement-batch.md`
  - **[N]**: 단계 3으로 진행 (그룹화 새로 시작).
  - **[X]**: HALT.

`{progress_file}`이 존재하지 않으면 단계 3으로 진행한다.


### 3. 미완료 항목 추출

`{task_file}`을 읽어 `- [ ]` 항목만 추출한다.
각 항목의 번호(앞의 숫자), 제목, 설명을 파악한다.
이미 `- [x]`로 표시된 항목은 무시한다.

미완료 항목이 없으면 HALT:
> 모든 리팩토링 항목이 완료되었습니다! 🎉 (`{task_file}`)


### 4. 항목 그룹화

`{grouping_strategy}` 설정을 기준으로 그룹화한다.

#### dependency 전략 (기본)

다음 기준으로 같은 배치로 묶는다:
- **동일 파일/클래스를 동시에 수정해야 하는 항목** — 한 파일에 두 항목이 함께 건드리면 하나의 배치
- **선행 조건 관계** — B가 A 완료 후에만 가능하면 순서는 유지하되 별도 배치
- **독립적인 항목** — 서로 다른 파일/관심사를 건드리면 각자 배치

#### layer 전략

- domain 레이어 변경 → 배치 1
- service 레이어 변경 → 배치 2
- controller/DTO 변경 → 배치 3
- test / 설정 변경 → 배치 4
- 여러 레이어를 동시에 수정해야 하는 항목은 영향 범위가 가장 큰 레이어 배치에 포함

#### manual 전략

모든 항목을 각각 개별 배치(1개씩)로 만든다.


### 5. 그룹화 결과 제시 및 승인 요청

다음 형식으로 출력한다:

---

**리팩토링 배치 계획** (`{task_file}`)

총 {N}개 배치 / {M}개 항목

**배치 1** — {배치 제목 (주요 관심사 요약)}
- #{번호} {항목 제목}
- #{번호} {항목 제목}
> 이유: {왜 함께 묶었는지 한 줄}

**배치 2** — {배치 제목}
- #{번호} {항목 제목}
> 이유: {왜 함께 묶었는지 한 줄}

... (이하 동일)

---

`[A]` 승인 — 이 순서대로 진행
`[E]` 수정 — 직접 그룹을 재구성해 주세요 (어떤 항목을 어떤 배치로 묶을지 알려주세요)
`[S]` 특정 배치만 — 몇 번 배치부터 시작할지 알려주세요

HALT하고 사용자 입력을 기다린다.


### 6. 사용자 입력 처리

- **[A]**: `{groups}`와 `{current_batch_index} = 0` 설정. 단계 7로 진행.
- **[E]**: 사용자가 재구성을 알려주면 `{groups}`를 재편성하고 단계 5로 돌아가 다시 제시.
- **[S]**: 시작 배치 번호를 `{current_batch_index}`로 설정. 단계 7로 진행.


### 7. progress 파일 생성

`{progression_dir}`이 존재하지 않으면 생성한다.

`.gitignore`를 확인해서 `progression/` 항목이 없으면 추가한다:
```
# iterative-refactor progress files
progression/
```

`{progress_file}`을 다음 형식으로 Write한다:

```markdown
---
task_file: {task_file}
current_batch_index: {current_batch_index}
started_at: {현재 ISO 8601 시각}
updated_at: {현재 ISO 8601 시각}
groups:
  - title: "{배치 1 제목}"
    items:
      - "#{번호} {항목 제목}"
      - "#{번호} {항목 제목}"
  - title: "{배치 2 제목}"
    items:
      - "#{번호} {항목 제목}"
---

# 진행 상황: {task_file basename}

총 {N}배치 / {M}항목 | 완료: 0/{N}

## 배치 1 — {제목} ⏳
- #{번호} {항목 제목}
- #{번호} {항목 제목}

## 배치 2 — {제목} ⏳
- #{번호} {항목 제목}

... (이하 동일)
```
