# Step 3: 설계 문서 초안 작성 → 검토 → 저장

## RULES

- 설계 문서는 반드시 사용자 `[A]` 승인 이후에만 파일로 저장한다.
- 정보가 부족한 섹션은 비워두지 말고 "추가 논의 필요" / "미정"으로 명시한다.
- **EARLY EXIT** = 이 step을 즉시 중단하고 지정된 파일로 이동한다.


## INSTRUCTIONS

### 1. 초안 작성

`{qa_log}`을 바탕으로 다음 형식의 설계 문서를 `{design_doc}`에 작성한다:

```markdown
# {topic 기반 제목}

## 개요

{한두 문단 요약}

## 배경 및 목표

{왜 필요한가, 무엇을 달성하려 하는가}

## 범위

### 포함

- {항목}

### 제외

- {항목}

## 요구사항

### 기능 요구사항

- {항목}

### 비기능 요구사항

- {항목 (성능, 호환성 등) — 정보 없으면 "미정"}

## 설계 개요

{아키텍처/주요 컴포넌트/데이터 흐름 — 정보 부족 시 "추가 논의 필요"로 표시}

## 영향 범위

- {영향받는 파일/모듈/시스템}

## 구현 작업 목록

- [ ] #1 {작업 항목}
- [ ] #2 {작업 항목}

## 리스크 및 미해결 질문

- {항목}
```

`구현 작업 목록`은 추후 다른 스킬이 체크리스트로 읽을 수 있도록 `- [ ] #{번호} {제목}` 형식을 유지한다.


### 2. 검토 요청

HALT하고 `{design_doc}` 전체를 출력한 뒤 제시한다:

> `[A]` 승인 — 이 내용으로 저장합니다.
> `[E]` 수정 — 변경하고 싶은 부분을 알려주세요.
> `[X]` 중단 — 저장하지 않고 종료합니다.

HALT하고 사용자 입력을 기다린다.


### 3. 사용자 입력 처리

#### [A] 승인

`{design_doc_path}`에 `{design_doc}`을 Write한다 (디렉토리가 없으면 생성).

> ✅ 설계 문서를 저장했습니다: `{design_doc_path}`
> "구현 작업 목록" 섹션은 이후 개발 스킬의 입력으로 사용할 수 있습니다.

HALT하고 종료한다.

#### [E] 수정

사용자의 피드백을 `{design_doc}`에 반영한다.
반영 후 단계 2로 돌아가 다시 제시한다.

#### [X] 중단

> 작업을 중단합니다. 설계 문서는 저장되지 않았습니다.

HALT.
