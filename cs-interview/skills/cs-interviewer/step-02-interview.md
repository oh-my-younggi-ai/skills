# Step 2: 면접 진행 — 질문 → 답변 → 키워드 → 꼬리 질문

면접의 핵심 루프. 한 개념씩, 메인 질문 → 답변 → 키워드 기반 꼬리 질문으로 깊이 파고든다.

## RULES

- **한 번에 한 질문, 질문 직후 턴 종료.** 질문 한 개를 출력하면 그 응답을 거기서 끝내고
  사용자의 실제 답변을 기다린다. **HALT는 "응답을 멈추고 사용자 입력을 받는다"는 뜻이다.**
- **사용자 답변을 절대 대필/추측하지 않는다.** 한 응답 안에 사용자의 답변을 지어내 채우거나,
  답변이 온 것처럼 가정하고 평가·다음 질문으로 넘어가서는 안 된다. 사용자 입력 메시지가
  실제로 도착했을 때만 다음 단계를 진행한다. (이걸 어기면 면접이 가짜가 된다.)
- 정답을 미리 다 알려주지 않는다. 면접관처럼 답변을 끌어내고 파고든다.
  (상세 평가·모범답안은 세션 끝 step-03에서 정리한다.)
- 평가 기준은 해당 위키 개념 페이지다. 페이지의 `# 핵심 개념`·`# 헷갈리는 점`을 정답/함정 기준으로 쓴다.
- `{wiki_root}`는 읽기만 한다. 면접 중 어떤 위키 파일도 수정하지 않는다.
- **EARLY EXIT** = 이 step을 즉시 중단하고 지정된 파일로 이동한다.

## INSTRUCTIONS — 현재 개념 1개 처리

### 1. 개념 페이지 로드 (읽기 전용)

`{concept}` = `{plan}[{current_index}]`. 위키에서 해당 개념 페이지를 읽어
`# 핵심 개념`, `# 면접 단골 질문`, `# 헷갈리는 점`을 파악한다.

### 2. 메인 질문

`# 면접 단골 질문`에서 하나를 고르거나, 없으면 `# 핵심 개념`에서 질문을 만든다.
질문만 제시한다 (몇 번째/총 몇 개인지 한 줄로 같이).

> **[{current_index+1}/{총 개수}] {주제}**
> {메인 질문}

**여기서 응답을 끝낸다(턴 종료).** 사용자의 실제 답변이 도착할 때까지 아무것도 더 출력하지 않는다.
답변이 오기 전에는 step 3 이후로 진행하지 않는다.

### 3. 답변 분석 — 키워드 추출

사용자 답변에서:
- **언급한 키워드/주장**을 뽑는다 (맞게 짚은 것, 모호한 것).
- 위키 `# 핵심 개념` 대비 **빠뜨린 핵심**, `# 헷갈리는 점` 대비 **건드린 함정/오해**를 파악한다.
- **사용자 답변 원문**과 직전 질문 원문을 `{transcript}`의 현재 개념 항목에 그대로 보존하고,
  분석 메모(강점/놓친 점/첨삭거리/꼬리거리)를 함께 누적한다. 원문은 요약하지 않는다 —
  step-03 보고서가 대화를 원문 그대로 옮기고 틀린 답변 아래 첨삭을 단다.

사용자가 "모르겠어요/패스"라고 하면: 짧은 힌트를 주고 다시 시도시키거나, 약점으로 기록하고 다음으로 넘어간다.

### 4. 꼬리 질문

키워드 분석에 근거해 꼬리 질문을 만든다 (개념당 최대 `{max_followups_per_concept}`개):

- **깊이 파기**: 답변이 언급한 키워드 중 핵심을 골라 "왜/어떻게"로 더 들어간다.
  (예: 답변에 "handshake"가 나오면 → "왜 2-way가 아니라 3-way인가요?")
- **함정 찌르기**: 위키 `# 헷갈리는 점`의 오해를 답변이 건드렸거나 비껴갔으면 그 지점을 검증한다.
- **빈 곳 메우기**: 핵심인데 답에서 빠진 부분을 한 번 더 물어 끌어낸다.

질문은 하나씩 — 꼬리 질문을 출력하면 **거기서 응답을 끝내고(턴 종료)** 사용자의 실제 답변을 기다린다.
답변이 도착한 뒤에만, 필요하면(그리고 한도 내에서) 다음 꼬리 질문으로 넘어간다.
여러 꼬리 질문을 한 응답에 몰아 쓰거나, 사용자 답변을 가정하고 이어가지 않는다.

면접 중 **위키에 없는 개념**이 답변에 등장하면 `{missing_concepts}`에 적어둔다 (지금 평가/정리하지 않음).
